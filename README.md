# Payroll Comparison Tool

This tool was built for a specific system but has been stripped of any names. This is not build for general use, just as a demonstration.

A tool for validating new payroll configuration during a client migration against a legacy payroll summary. Upload CSV exports from a legacy payroll system and from the new system, map columns between them, and review per-employee differences to confirm the new system is producing equivalent output before go-live.

## Use case

Implementors run the payroll comparison during the transition period. The implementor will:

1. Create a **Comparison** for the pay period.
2. Upload Legacy and new system CSV exports as Sources.
3. Map columns (New columns auto-load by category; Legacy columns are mapped manually or via AI suggestion).
4. Run the comparison and review per-employee **Differences** by category (Earnings, FICA, Deductions, etc.).
5. Mark discrepancies resolved (with optional notes), or investigate them in the payroll system.
6. Export results as CSV, Excel (static or dynamic/formula-driven), or Google Sheets.

## Getting started

**1. Install dependencies**

```bash
bun install
```

**2. Configure environment**

Copy the variables below into a `.env` file:

```
ANTHROPIC_API_KEY=
BETTER_AUTH_SECRET=
URL=http://localhost:5173
PORT=5173
GOOGLE_CLIENT_ID=
GOOGLE_CLIENT_SECRET=
```

Or run `bun run setup` to auto-detect a free port and write `URL` and `PORT` for you.

`BETTER_AUTH_SECRET` can be any long random string. `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` come from a Google Cloud OAuth 2.0 credential with `http://localhost:5173/api/auth/callback/google` as an authorized redirect URI.

**3. Apply local D1 migrations**

```bash
bunx wrangler d1 migrations apply comparison_db --local
```

**4. Start the dev server**

```bash
bun run dev
```

The app is served at `http://localhost:5173`. Hot module reload is enabled via Vite.

## Developer flow

**Run tests**

```bash
bun test
```

Tests live alongside their modules in `__tests__/` subdirectories. The comparison engine, export services (Excel + Sheets), and mapping logic all have unit test suites.

**Generate a new migration**

After editing `src/server/db/schema/schema.ts`, generate and apply locally:

```bash
bun run db:generate
bunx wrangler d1 migrations apply comparison_db --local
```

Migrations run automatically via a separate GitHub Actions workflow when `.sql` files under `drizzle/` are pushed to `main`. Only add backward-compatible migrations (new columns with defaults, new tables). Destructive changes (dropping columns, renaming) must go in a separate PR after all code referencing the old schema is already merged and deployed.

**Deploy**

```bash
bun run deploy
```

This runs `vite build && wrangler deploy`. Remote D1 migrations are applied automatically by the GitHub Actions workflow on push to `main`.

**Architecture decisions**

Design decisions, rejected alternatives, and their rationale are documented in [`docs/adr/`](docs/adr/). Read these before making changes to auth, the comparison engine, export formats, or the column-mapping flow.
